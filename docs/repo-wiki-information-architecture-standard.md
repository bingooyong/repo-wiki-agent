# Repo Wiki 信息架构标准

本文定义 repo-agent 生成 Qoder-like Repo Wiki 时必须遵守的文档体系结构。生成器不得直接把目录、接口、DTO、Entity、测试文件按扫描结果平铺成页面；所有内容必须先归入稳定的信息架构，再由证据和 LLM 生成可阅读的专题文档。

## 目标

repo-agent 的 Repo Wiki 面向工程读者，而不是面向扫描器输出。读者进入 Wiki 后，应能按“理解项目 -> 理解架构 -> 理解服务 -> 理解数据 -> 理解 API -> 开发与运维 -> 故障处理”的路径阅读。

生成结果必须满足三个目标：

- 可导航：左侧目录是项目知识地图，不是文件列表。
- 可阅读：每页是有上下文、有解释、有源码证据的文章，不是清单导出。
- 可维护：页面与源码、commit、证据、质量门禁关联，支持增量更新。

## 顶层目录标准

| 顶层目录 | 目的 | 典型读者 | 是否必需 |
|---|---|---|---|
| 项目概述 | 解释项目定位、背景、核心能力、快速开始、模块组织 | 新成员、管理者、评审者 | 必需 |
| 架构设计 | 解释系统边界、整体架构、微服务/模块关系、数据流、集成关系 | 架构师、核心开发 | 必需 |
| 核心服务 | 解释核心业务服务、职责边界、调用关系、关键流程 | 后端开发、维护者 | 必需 |
| 运行时服务 | 解释 Python 服务、Agent、Worker、前端应用等非核心后端服务 | 服务负责人 | 按项目可选 |
| 数据模型 | 解释核心实体、服务数据模型、数据库架构、迁移策略 | 后端开发、DBA、数据治理 | 必需 |
| API参考 | 解释 API 分组、调用约定、认证授权、错误处理、核心服务 API | 前后端开发、集成方 | 必需 |
| 开发指南 | 解释本地开发、代码规范、测试、贡献流程、性能调优 | 开发者 | 必需 |
| 部署运维 | 解释环境、配置、部署、监控、日志、备份、扩缩容 | 运维、SRE | 必需 |
| 操作手册 | 解释日常操作、任务流程、管理后台、批处理、人工介入步骤 | 运营、支持、运维 | 项目有运行流程时必需 |
| 安全合规 | 解释认证、授权、数据保护、审计、供应链、安全扫描 | 安全、审计、开发 | 必需 |
| 故障排除与维护 | 解释常见故障、排查路径、健康检查、回滚、恢复 | 维护者、SRE | 必需 |

## 页面类型标准

### 总览页

总览页用于建立阅读框架，不承载细节堆叠。

必备章节：

- 简介
- 目录
- 项目/领域定位
- 核心能力
- 子主题导航
- 关键源码证据
- 后续阅读路径

### 架构页

架构页必须解释设计意图和边界，不允许只列目录。

必备章节：

- 架构目标
- 系统边界
- Mermaid 架构图
- 核心组件职责
- 数据流或调用流
- 部署/运行时关系
- 设计取舍
- 源码证据

### 服务页

服务页必须从业务职责和运行边界解释服务。

必备章节：

- 服务概述
- 业务职责
- 入口模块
- 核心组件
- 依赖关系
- API/事件/任务入口
- 数据模型关联
- 故障排查
- 源码证据

### API页

API 页必须按服务族、调用场景、认证和错误约定聚合。默认禁止生成单 endpoint 页面。

允许页面：

- API参考
- 核心服务API
- Python服务API
- API网关API
- Agent代理API
- 前端应用API
- 认证授权API
- 错误处理与状态码
- 单个服务族 API，例如“契约管理服务API”

禁止默认生成：

- `GET /`
- `DELETE /{id}`
- `POST /xxx`
- 每个 endpoint 一个 Markdown

只有用户显式启用 endpoint 明细模式时，才能生成 endpoint 清单，并且必须放在服务族页内部，不进入主导航。

### 数据模型页

数据模型页必须按核心实体、服务模型、数据库结构、迁移策略组织。默认禁止 DTO/Request/Response/Entity 原始类页面进入主导航。

允许页面：

- 数据模型
- 核心数据模型
- 服务数据模型
- 数据库架构
- 数据迁移策略
- 单个服务的数据模型专题
- 重要业务实体专题

禁止默认生成：

- 每个 DTO 一个页面
- 每个 Request/Response 一个页面
- 每个 Entity 无上下文单页
- 原始字段清单页

### 开发指南页

开发指南必须回答“如何在这个项目上开发”，不是 README 复制。

必备主题：

- 本地环境搭建
- 代码结构约定
- 测试策略
- 调试方式
- 代码规范
- 提交流程
- 性能调优

### 部署运维页

部署运维必须回答“如何运行和维护这个系统”。

必备主题：

- 环境与配置
- 部署拓扑
- 启停流程
- 日志与监控
- 数据备份
- 扩缩容
- 回滚策略

### 操作手册页

操作手册面向日常使用和人工介入流程。没有明确运营流程的项目可不生成。

必备主题：

- 常用操作
- 管理任务
- 批处理任务
- 巡检清单
- 异常处理
- 权限与审计注意事项

## Qoder 对照结论

AI_API_Atlas 的 Qoder Repo Wiki 体现了三个关键能力：

- 顶层分类稳定：项目概述、架构设计、核心服务、Python服务、数据模型、API参考、开发指南、安全合规、故障排除与维护等。
- 专题层级明确：API 按核心服务 API、Python 服务 API、认证授权、错误处理分组；数据模型按核心数据模型、服务数据模型、数据库架构分组。
- 页面不是扫描清单：大多数页面围绕业务能力、服务职责、数据语义、调用流程组织，而不是围绕文件路径或类名组织。

repo-agent 必须把上述能力作为生成标准，而不是事后用 verify 补救。

## 生成器规则

生成器必须按以下顺序工作：

1. 识别项目类型、业务域、运行时类型和主要服务。
2. 构建固定信息架构树。
3. 将模块、API、数据模型、配置、脚本归入信息架构节点。
4. 为每个节点选择证据，而不是为每个证据创建页面。
5. 使用 LLM 生成专题文章。
6. 写入 Qoder-like 中文目录结构。
7. 生成 manifest navigation tree。
8. 执行 strict gate 与 Qoder baseline 对比。

## 质量门禁

以下情况必须判定为 `NOT_READY`：

- 主导航出现 raw endpoint 页，例如 `GET /`、`DELETE /{id}`。
- 主导航出现 raw DTO/Request/Response/Entity 页。
- API 页面以 endpoint 表格为主体。
- 数据模型页面以字段列表或类名列表为主体。
- 页面缺少 prose、目录、源码引用或 Mermaid 图。
- 页面没有解释业务职责、调用边界或维护风险。
- 输出目录不是 Qoder-like 中文目录结构。

## 当前 repo-agent 差距

当前 repo-agent 已修复 endpoint 和 raw model dump 的一部分问题，但仍存在两个系统性差距：

- 输出路径仍偏技术路径，例如 `pages/api/*.md`，没有完全写成 `API参考/核心服务API/*.md` 的中文嵌套目录。
- 页面规划仍混合了“模块页”和“专题页”，需要进一步把模块清单折叠为服务专题和能力专题。

下一阶段应优先实现中文嵌套 layout writer 和信息架构驱动 planner，而不是继续微调单页模板。
