# Repo-Wiki 对齐 Qoder 风格后续任务规划

文档属性：任务规划  
前置文档：`docs/qoder-repo-wiki-design-analysis.md`  
目标：把当前 `repo-agent` 的输出从“结构化导出器”提升为“文档中心型知识库生成器”

## 1. 规划原则

后续任务按以下原则执行：

- 先修信息架构，再修文本模板
- 先补 section 层，再优化模块页
- 先让总览文档可读，再追求细节覆盖
- 先把内容聚合做对，再考虑模型调用增强
- 每个任务都必须有明确验收条件，避免再次出现“文件生成了但不能读”

## 2. 后续任务列表

### 任务 1：重构文档输出契约，新增文档中心层

目标：

- 在现有 `docs/00~05` 和 `docs/modules/` 之间新增 `docs/sections/` 作为专题层
- 明确三层输出契约：总览层、专题层、模块层

交付物：

- 新的文档 contract 定义
- `docs/sections/` 目录规划
- section 级模板占位

验收标准：

- 生成器能稳定输出 `docs/sections/project`、`architecture`、`services`、`data-model`、`api`、`operations` 等目录
- `verify` 能识别 section 层是否齐全

### 任务 2：实现业务域分类器

目标：

- 不再直接把物理目录当成一级知识模块
- 为模块分配业务域、服务族、运行角色

交付物：

- 域分类规则
- 模块到业务域映射产物
- 分类回退策略

验收标准：

- `module-index` 中每个模块都能映射到至少一个高层域
- `03-module-map` 可按业务域输出，而不是按目录平铺

### 任务 3：重写 `00-overview` 生成逻辑

目标：

- 让 `docs/00-overview.md` 从列表页变成介绍页

最低章节要求：

- 项目定位
- 核心问题
- 核心能力
- 快速开始
- 阅读导航

验收标准：

- 文档中至少有 5 个固定章节
- 每个章节至少有 1 段完整文字说明
- 不允许只有清单和统计数字

### 任务 4：重写 `01-architecture` 生成逻辑

目标：

- 输出真正的系统设计说明，而不是模块和 API 摘抄

最低章节要求：

- 系统分层
- 服务协作关系
- 核心数据流
- 存储与检索设计
- 增量更新与治理闭环

验收标准：

- 文档必须包含 Mermaid 架构图
- 必须显式解释 `.repo-wiki`、`source-of-truth`、`docs` 三层关系

### 任务 5：重写 `03-module-map`，改为“域地图”

目标：

- 从模块列表升级为领域地图

交付物：

- 核心服务分组
- Python 服务分组
- 前端与工作台分组
- 支撑与治理模块分组

验收标准：

- 输出至少包含 3 组以上高层分类
- 每组下展示模块职责、上下游关系和入口文档

### 任务 6：重写 `04-api-contracts` 的聚合方式

目标：

- 不能再全量平铺 endpoint
- 改为按服务族、认证方式、主题域聚合

交付物：

- API 分组器
- 认证与状态码摘要
- 重点入口 API 清单

验收标准：

- 单文档不得输出全量原始 endpoint 长列表
- 必须包含“服务/API 分组”与“调用约定”两个固定章节

### 任务 7：重写 `05-data-model`，加入去重与领域聚合

目标：

- 不能再把数百上千个模型直接展开
- 需要提取核心域模型和服务专属模型

交付物：

- 模型去重规则
- 核心实体识别规则
- 数据库与迁移摘要

验收标准：

- 文档必须分为“核心数据模型”“服务数据模型”“数据库/迁移策略”三段
- 大量重复 `Builder`、`Entity` 类名不得直接污染总览页

### 任务 8：为 `docs/sections/` 实现专题页生成

目标：

- 让截图中的导航结构可由 repo-wiki 稳定生成

建议优先专题：

- `project/项目概述.md`
- `architecture/架构设计.md`
- `services/核心服务.md`
- `services/python-services.md`
- `data-model/核心数据模型.md`
- `api/API参考.md`
- `operations/部署运维.md`
- `development/开发指南.md`
- `security/安全合规.md`
- `operations/故障排查与维护.md`

验收标准：

- 至少 8 个专题页生成成功
- 专题页之间有相互导航链接

### 任务 9：升级 `verify --ci` 为内容质量校验

目标：

- 校验从“存在性”升级为“存在性 + 内容质量”

新增检查项：

- 总览页最小章节数
- 总览页最小段落数
- section 层完整性
- API/DataModel 总览是否经过聚合
- 模块页是否挂接到 section/总览导航

验收标准：

- 缺少文字描述时必须 `FAIL` 或至少 `WARN`
- 空模板或纯列表模板不能通过质量校验

### 任务 10：建立 qoder 基线对比回归

目标：

- 把 qoder 风格输出固化为回归基准

交付物：

- 基线仓库样本
- 结构对比脚本
- 内容质量对比清单

验收标准：

- 每次生成后可以比较目录层级、章节数量、导航完整性、聚合质量
- 能明确告诉我们“比 qoder 少了什么”

## 3. 执行顺序建议

建议按下面顺序推进：

1. 任务 1
2. 任务 2
3. 任务 3
4. 任务 4
5. 任务 5
6. 任务 6
7. 任务 7
8. 任务 8
9. 任务 9
10. 任务 10

原因很简单：

- 先定结构
- 再定分类
- 再重写五个核心总览页
- 最后补专题页和治理闭环

## 4. 第一批必须先做的任务

如果只做最关键的一批，优先级如下：

- P0：任务 1，任务 2，任务 3，任务 4
- P1：任务 5，任务 6，任务 7
- P2：任务 8，任务 9，任务 10

第一批完成后，至少能解决你这次明确指出的三个核心问题：

- 文档没有文字描述
- 文档没有合理目录规划
- 输出没有 qoder 风格的文档中心感

## 5. 里程碑判断

达到“接近 qoder 风格”的最低标准，应满足以下条件：

- `docs/00-overview.md` 和 `docs/01-architecture.md` 可以直接给新人阅读
- 左侧导航可以从 `docs/sections/` 稳定生成
- `03-module-map`、`04-api-contracts`、`05-data-model` 都完成聚合而不是平铺
- `verify --ci` 会对内容质量给出明确反馈

在达到这些标准之前，不建议继续把当前模板体系扩展到更多仓库。
